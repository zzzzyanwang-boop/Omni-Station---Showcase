# Publication Boundary

The public showcase is generated by whitelist, not by filtering the private repository.

## Allowed

- Architecture summaries.
- Sanitized diagrams.
- Source-shaped placeholder files that preserve public-safe real paths and filenames while replacing implementation bodies with summaries.
- Redacted capability placeholders for sensitive private surfaces where real filenames are not safe to publish.
- Synthetic examples with toy symbols and fake hashes.
- General engineering decision records.

## Forbidden

- Production source bodies from the private repository.
- Real configuration, task state, handoff state, run logs, or queue state.
- Real artifacts, model output, feature matrices, labels, OOF predictions, or replay reports.
- Any reference to real local filesystem paths, private usernames, or internal runtime roots.
- Any strategy detail that would make the private research reproducible.
- Real private filenames when the filename itself reveals research direction, execution posture, vendor detail, or unpublished results.

## Release Checklist

1. Build a new export directory from public-safe templates and a whitelist of allowed source paths.
2. Replace implementation bodies with placeholders before any publication step.
3. Scan for private paths, secrets, real artifact extensions, implementation tokens, and internal identifiers.
4. Initialize a new git repository in the export directory only.
5. Push the export repository, never the private repository or its history.
6. Re-scan the published repository after push.
